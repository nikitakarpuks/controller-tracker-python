  ---
  Step 2 — _visibility.py: Two geometry-transform helpers

  Placed alongside the existing _rays_blocked_by_* functions:

  _box_to_cam(box: Box3D, R: np.ndarray, t: np.ndarray) -> Box3D
  Returns a new Box3D instance. center_new = R @ box.center + t, axes_new = R @ axes_matrix (each column rotated), half_dims unchanged.

  _cylinder_to_cam(cy: Cylinder3D, R: np.ndarray, t: np.ndarray) -> Cylinder3D
  Returns a new Cylinder3D instance. center_new = R @ cy.center + t, axis_new = R @ cy.axis, all scalar fields unchanged.

  Both return new instances — originals are never mutated, since the same geometry objects are reused every frame for self-occlusion checks.

  ---
  Step 3 — _visibility.py: 2D projection gate

  _occlusion_gate(tvec_A, tvec_B, br_A, br_B, focal_px, margin_px) -> bool

  Projects both controller origins with simplified pinhole (x_proj = tvec[0]/tvec[2] * focal). Returns True (gate open, run the full check) when:

  dist_2d(proj_A, proj_B)  <  focal * (br_A / tvec_A[2]  +  br_B / tvec_B[2])  +  margin_px

  Hard-return False immediately if either tvec[2] <= 0. Pinhole projection is fine here — the gate is intentionally conservative and margin_px
  absorbs any distortion-induced error.

  ---
  Step 4 — _visibility.py: _cross_occluded_mask

  def _cross_occluded_mask(
      R_B, tvec_B,       # T_cam_B: B's current predicted pose in this camera
      positions_B,       # (N, 3) B's LED positions in controller-B frame
      R_A, tvec_A,       # T_cam_A: A's current-frame pose in this camera
      geom_A,            # ControllerGeometry of the occluder (controller A)
      br_A, br_B,        # bounding radii for gate
      focal_px,
      gate_margin_px,
  ) -> np.ndarray        # bool (N,), True = LED occluded by A's body

  Execution order:

  1. Call _occlusion_gate → return np.zeros(N, bool) immediately if gate fails.
  2. Transform A's primitives to camera frame using _box_to_cam / _cylinder_to_cam with R_A, tvec_A.
  3. Transform B's LEDs to camera frame: led_cam_B = (R_B @ positions_B.T).T + tvec_B.

  5. blocked = np.zeros(N, bool). Iterate A's transformed boxes and cylinders, OR-accumulate: blocked |= _rays_blocked_by_box(cam_origin, led_cam_B,

  6. Return blocked.

● Good question, worth thinking through carefully.

  Yes, theory + data fitting is the right approach, and here's why it's not just "the only way" but genuinely the correct engineering choice:

  The functional form is already known from physics:
  - Blob area ∝ cos(θ) / d² — projected solid angle of a tilted LED disk
  - Blob brightness ∝ cos(θ) / d² — Lambertian emission + inverse-square falloff

  Your existing model a/d² + b is already this, just without the cos(θ) factor (because you only had per-controller depth, so angle variation was averaged out). The extension is a *
  facing_cos / d² + b for each LED individually.

  Why not pure data-driven?
  You'd need a massive calibration dataset covering the full (d, θ) space, and the result would be a lookup table that doesn't generalize well to new camera positions or LED layouts.

  Why not pure theory?
  Real LEDs aren't ideal Lambertian emitters, camera sensors are non-linear, lenses have falloff. The a and b parameters absorb all that.

  What you already have:
  get_predicted_led_projections_per_camera already returns [proj_x, proj_y, depth_m, facing_cos] per visible LED. Both d and cos(θ) are right there. The calibration CSV just needs to
  record per-LED (not per-controller) observations: (d_i, facing_cos_i, brightness_i, area_i) during frames where tracking is confirmed good — then re-fit a * facing_cos / d² + b.

  The harder problem is the one you pointed out: blob detection currently runs globally with scalar thresholds before any LED assignment is known. The per-LED thresholds only become
  usable after you have a predicted pose. So realistically you end up with two regimes:
  1. Cold start / no pose: global thresholds as now
  2. Warm / pose available: per-LED local search with thresholds computed from predicted (d_i, facing_cos_i) — this is the pose-guided path you already partially have

  That's not a limitation, it's the natural architecture. Does this align with what you were thinking?

  Brute search — cold start

  Each PoseSearcher is bound to one camera (self.camera, self._cam). In cold start, brute_search is called once per camera, each working purely on its own blobs. other_cameras_blobs is

● The idea is solid. Here's how I think about it:

  What you gain from locked-pair PnP:

  The locked pairs are unambiguous and already known — running solvePnP on them
  costs one call but gives you a pose that's strictly tighter than rvec_pred.
  Since proximity is only called when tracking is healthy, you'll typically have
  4-8 locked pairs, and that PnP will converge fast from the predicted pose as
  initial guess.

  Two downstream effects, in order of impact:

  1. Candidate tightening (high leverage). Re-project hyp_k LEDs with the
  refined pose and rebuild their candidate blobs using a tighter radius — say
  the prox_reproj_px threshold rather than expansion_px. If a hyp LED currently
  has 3 candidates and tightening drops it to 1, that LED becomes locked too,
  eliminating it from the combo enumeration entirely. The combo count is
  multiplicative, so even halving candidates per LED can cut the total by orders
  of magnitude.
  2. Better initial guess in _score_hyp (lower leverage). Using the refined pose
  as useExtrinsicGuess means ITERATIVE PnP converges in fewer iterations per
  combo. Helpful but secondary — if the candidate tightening already collapsed
  hyp_k from 5 LEDs to 2, this doesn't matter much.

  One risk to be aware of:

  If the predicted pose has enough drift that some locked pairs are actually
  mis-assigned (wrong unique blob), the locked-pair PnP will produce a slightly
  wrong refined pose. The safeguard here is checking the reprojection error on
  the locked pairs after solving — if it's unexpectedly large (say >
  prox_reproj_px * 2), fall back to rvec_pred and skip candidate tightening.
  This keeps the existing logic as the safe fallback.

  Verdict: The candidate tightening is where the real speedup lives and it
  directly addresses the "brute can outperform proximity" problem. The current
  code's sorting of 0-None combos by _raw distance already exploits the
  predicted pose ordering, but if you start the hyp enumeration with fewer
  candidates to begin with, you never generate the bad combos at all. Worth
  implementing.
